# E-commerce API Test Examples: Guest & User Checkout Flow

This file contains example API requests to test the complete e-commerce flow that supports both guest and authenticated users. The flow ensures orders are only created after successful payment and shipment.

## 🛒 New Flow Overview

1. **User selects products** - Browse and add items to cart
2. **Cart in session** - Cart stored in session for guests, database for users
3. **Login/Register (optional)** - If user chooses to login/register, migrate cart to database
4. **Manage addresses (optional)** - Authenticated users can save/manage multiple addresses
5. **Create shipment** - Generate shipment through Bosta API (uses stored or provided address)
6. **Process payment** - Initiate payment through Paymob (uses stored or provided address)
7. **Create order** - Order is only created after BOTH shipment and payment are successful

## Prerequisites
- Run `php artisan db:seed` to populate the database with test data
- Replace `BASE_URL` with your API base URL (e.g., http://localhost:8000/api)
- No authentication required for cart operations (guest flow supported)
- Authentication optional for checkout flow

```bash

  -d '{
    "shipping_address_id": 1,
    "billing_address_id": 1,
    "notes": "Please handle with care"
  }'
```


## 🛒 Complete Flow: Add to Cart → Register/Login → Checkout → Order

This section demonstrates the complete end-to-end flow starting from adding products to cart as a guest, then registering or logging in to save the cart data in the database, and completing the order.

### Step 1: Browse Products (As Guest)
```bash
curl -X GET BASE_URL/products
```

### Step 2: Add Products to Cart (As Guest - Stored in Session)
```bash
# Add Wireless Headphones
curl -X POST BASE_URL/cart/add \
  -H "Content-Type: application/json" \
  -d '{
    "product_id": 1,
    "quantity": 2
  }'

# Add Cotton T-Shirt
curl -X POST BASE_URL/cart/add \
  -H "Content-Type: application/json" \
  -d '{
    "product_id": 3,
    "quantity": 1

```

```bash

  -H "Content-Type: application/json" \

    "password_confirmation": "password123"
  }'
```

**Note:** Behind the scenes, the system automatically calls `SessionCartService->migrateSessionToDatabase()` to transfer all guest cart items to the user's database cart.

## Flow Order Summary

1. **Browse Products** - GET /products (as guest)
2. **Add to Cart** - POST /cart/add (stored in session as guest)
3. **Review Cart** - GET /cart (shows is_guest: true)
4. **User Registration** - POST /register (automatically migrates session cart to database)
5. **Cart Migration** - Session cart automatically transferred to user database cart
6. **Manage Addresses** - POST /addresses (create shipping/billing addresses)
7. **Review Cart Again** - GET /cart (now shows is_guest: false)
8. **Initiate Checkout** - POST /checkout/initiate (creates shipment + payment using stored addresses)
9. **Complete Payment** - User pays via Paymob iframe
10. **Payment Webhook** - Paymob sends success webhook
11. **Order Creation** - POST /checkout/complete creates final order
12. **Order Management** - User can view/track orders

## 📍 Address Integration Scenarios

### Scenario 1: Guest Checkout (No Addresses Stored)
- User provides shipping/billing addresses inline during checkout
- Addresses used directly for Bosta and Paymob APIs
- No address storage in database

### Scenario 2: Authenticated User - First Time
- User logs in/registers
- Creates addresses during or after registration
- Uses stored addresses for checkout
- Addresses saved for future use

### Scenario 3: Authenticated User - Returning Customer
- User logs in (cart migrates automatically)
- Selects from saved addresses or creates new one
- Uses stored addresses for faster checkout
- Default address pre-selected

### Scenario 4: Mixed Addresses
- User uses stored address for shipping
- Provides different billing address inline
- System handles both stored and inline addresses seamlessly

### Address API Integration
- **Bosta API**: `toBostaAddress()` formats address for shipment creation
- **Paymob API**: `toPaymobAddress()` formats address for payment processing
- **Order Storage**: Addresses stored in order with proper relationships
- **Validation**: Addresses validated for required fields per API requirements

## 🔧 Error Scenarios

### Cart Errors
- `422` - Product not available/insufficient stock
- `404` - Product not found

### Checkout Errors
- `422` - Cart empty, validation failed, shipment/payment failed
- `500` - Server error during checkout process

### Payment Errors
- `422` - Payment initiation failed
- `500` - Webhook processing error

### Step 3: Simulate Payment Failure via Callback
```bash
curl -X GET "BASE_URL/payment/callback?success=false&transaction_id=txn_failed_123&order_id=TEMP-FAILURE-123"
```

**What happens internally:**
1. Payment callback receives `success=false`
2. System detects payment failure
3. **Shipment termination triggered**: `BostaApiService::cancelDelivery()` called with terminate endpoint
4. API call: `POST /deliveries/business/BOSTA-FAILURE-123/terminate`
5. Logs show termination attempt and result
6. Pending checkout data remains (no order created)

**Expected Response:**
```json
{
  "success": false,
  "payment_status": "failed",
  "order_id": "TEMP-FAILURE-123",
  "payment_id": "txn_failed_123",
  "tracking_number": "BOSTA-FAILURE-123"
}
```

## 🔍 Payment Failure Flow Details

### **Shipment Termination Process**
1. **Trigger**: Payment failure detected (callback/webhook)
2. **API Call**: `POST /deliveries/business/{trackingNumber}/terminate`
3. **Success**: Shipment marked as terminated/cancelled in Bosta
4. **Failure**: Logged as error, shipment may remain active
5. **Cleanup**: Pending checkout data removed, session cleared

### **Key Behaviors**
- ✅ **No order created** on payment failure
- ✅ **Shipment terminated** using official Bosta terminate endpoint
- ✅ **Stock not reduced** (no order items created)
- ✅ **Cart preserved** for retry attempts
- ✅ **Session cleaned** after failure handling
- ✅ **Comprehensive logging** for debugging

### **Error Scenarios**
- **Terminate API fails**: Shipment remains active, logged as error
- **Invalid tracking number**: Exception thrown, logged
- **Network issues**: Retry logic could be added in future
